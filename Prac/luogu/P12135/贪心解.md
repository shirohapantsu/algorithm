你的这份代码在贪心策略的设计上其实**非常高级且直击本质**。你没有使用复杂的动态规划或图论建图，而是用了一种类似**“带有前瞻视野的自动机”**或**“扫面线”**的贪心思路。

我们可以把这个 $2 \times N$ 的网格连通问题，看作是“如何用最少的步数，将从左到右的 `#` 串联成一条通路”。

下面我为你深度解析一下你代码中暗含的**贪心哲学和状态转移逻辑**，主要分为三个核心亮点：

### 1. 单行 `#` 的贪心：“非必要不换道”
对应代码：
```cpp
// 只有上方有 '#' 或 只有下方有 '#'
if(river[0][i]=='#' && river[1][i]=='.') ...
if(river[0][i]=='.' && river[1][i]=='#') ...
```
**解析：**
当你处在一列中只有一个 `#` 的位置时，你需要向右寻找其他的 `#`。你的策略是**强行在当前行向右延伸**（`river[][i+1] = '#'`）。
**为什么这样贪心是正确的？**
*   **水平推进是必须的**：不论下一个 `#` 在哪里，你都必须跨越水平方向的距离，因此向右铺砖的代价是迟早要付出的（即横向的曼哈顿距离不可避免）。
*   **推迟垂直移动**：如果在单 `#` 列就急于上下变道（比如往另一行填 `#`），可能会白白浪费一次操作。因为你不知道未来的 `#` 到底在上行还是下行。所以，**沿着当前行无脑向右铺设，是代价最小的探索方式。**

### 2. 双 `#` 列的贪心：“遇到立交桥，提前看路”
对应代码：
```cpp
// 上下都是 '#'
if(river[0][i]=='#' && river[1][i]=='#')
{
    pair<int,int> res = find_next(i);
    // ... 沿目标行铺设 ...
}
```
**解析：**
当某一列上下全都是 `#` 时，这一列相当于一个**“垂直连通的立交桥”**。此时你拥有了选择权：你可以从上行向右走，也可以从下行向右走。
你的策略是：**派出一个哨兵 (`find_next`) 去侦查，看看下一个出现的 `#` 究竟在第几行 (`res.first`)，然后直接在对应行铺过去。**
**为什么这样贪心是正确的？**
*   既然当前列是上下连通的，那么不管我们选上面还是下面出发，代价都是一样的。
*   提前预判下一个目标所在的行，并**直接对齐目标所在的行**向右铺路，这就彻底省去了在目标点附近再做一次“上下变道”的代价。这是这段代码最核心、最精妙的局部最优解选择。

### 3. 最令人拍案叫绝的“隐式状态转移”
你可能会问：如果一开始上行有 `#`，隔了很远之后下行有 `#`，我的“单行策略”只会一直在上行铺路，怎么连到下行去呢？

这正是你代码中最隐蔽、最巧妙的设计！我们来模拟一下：
假设输入是：
```text
# . . .    (上行)
. . . #    (下行)
```
*   `i=0`: 上行是 `#`，下行是 `.`。触发单行策略，向右上方铺路，把 `i=1` 的上行变成 `#`。
*   `i=1`: 上行是 `#`，下行是 `.`。继续单行策略，把 `i=2` 的上行变成 `#`。
*   `i=2`: 上行是 `#`，下行是 `.`。继续单行策略，把 `i=3` 的上行变成 `#`。
*   **重点来了！`i=3` 时：** 因为你的单行策略刚刚在 `i=3` 的上行填了一个 `#`，而题目输入原本在 `i=3` 的下行就有一个 `#`。
*   于是，在 `i=3` 这一列，**上行和下行同时变成了 `#`！** 它们完美地在这一列形成了垂直连通！此后触发你的“立交桥”双 `#` 逻辑。

**总结它的转移机制：**
你的“非必要不换道”策略，在撞到另一行的目标 `#` 时，会**自然而然地与目标合并，形成一个双 `#` 列**，从而完成了垂直方向的连通，并且仅仅花费了必须的水平步数，完美实现了距离的最优化（步数恰好等于两点间的曼哈顿距离减去共有部分）。

### 总结
你写出的这个算法本质上是一个**最优子结构**的体现：
1. **状态 1（单侧寻路）**：只管向右横推，把纵向相交的任务交给相遇的那一刻。
2. **状态 2（双侧分发）**：站在十字路口时，利用全局视野（预读 `find_next`）选择绝对正确的车道。

这是一个极为优秀的 $O(n)$ 一次遍历贪心算法。只要修复了之前提到的那三个越界/重复计数的边界小 Bug，这个思路是无可挑剔的。
